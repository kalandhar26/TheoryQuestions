# Objects, Properties, and Methods in PowerShell

## Everything in PowerShell is an Object

In PowerShell, every piece of data — whether it is a process, a file, a service, or a date — is an **object**. And every
object has a **type** that defines what it is and what it can do.

To explore an object's type, properties, and methods, pipe it into `Get-Member`:

```powershell
Get-Process chrome | Get-Member
```

The `TypeName` field at the top of the output tells you exactly what kind of object you are working with. The rest of
the output lists all the **properties** (data the object holds) and **methods** (actions the object can perform)
available to you.

### Namespaces

Objects belong to **namespaces** — large containers that group related types and functionality together. Namespaces can
be further organized into **sub-namespaces**, creating a structured hierarchy that keeps the vast .NET ecosystem
organized and navigable.

---

## Viewing Object Properties

Understanding how to view object properties is a huge step forward in your PowerShell journey — it opens up an enormous
range of possibilities for managing files, processes, services, and more.

To access a property, wrap the command in parentheses and use dot notation:

```powershell
(Get-Process chrome).CPU
(Get-Process chrome).Company
(Get-Process chrome).Name
(Get-Process chrome).StartTime
(Get-Process chrome).WorkingSet
```

You can also perform calculations directly on numeric properties:

```powershell
(Get-Process chrome).WorkingSet / 1024    # Convert bytes to kilobytes
(Get-Process chrome).WorkingSet / 1MB     # Convert bytes to megabytes
```

### Why This Matters — A Practical Example

Suppose you need to find all files larger than 1 GB in a directory, or all files created in the last 24 hours. Rather
than guessing or using slow manual searches, you can filter directly on object properties like `Length` (file size) and
`CreationTime` — making these tasks precise, fast, and scriptable.

## Modifying Object Properties

Properties are not just for reading — you can write to them too. This is where PowerShell becomes genuinely powerful for
automation.

**Real-world scenarios where this matters:**

- You are a system administrator who needs to update the `LastWriteTime` on a batch of files to comply with a new data
  retention policy.
- You are a developer who needs to clear the read-only status on multiple files before your application can process
  them.

### Modifying `LastWriteTime`

```powershell
# View the current value
(Get-ChildItem file1.txt).LastWriteTime

# Assign a new date (simple string — works in straightforward cases)
(Get-ChildItem file1.txt).LastWriteTime = "01/01/2025"

# Cast explicitly to datetime for clarity
(Get-ChildItem file1.txt).LastWriteTime = [datetime]"01/01/2025"

# Parse with an exact format — essential for international dates or precise timestamps
(Get-ChildItem file1.txt).LastWriteTime = [datetime]::ParseExact("01/01/2025", "MM/dd/yyyy", $null)
```

> **When working with dates from different countries or timestamps with milliseconds, `ParseExact` becomes essential.**
> It removes any ambiguity about how PowerShell should interpret the string — you define the exact format, and
> PowerShell
> follows it precisely.

### Modifying `CreationTime` with a Full Timestamp

```powershell
# View the current CreationTime
(Get-ChildItem file1.txt).CreationTime
# Output: 12 December 2024 00:00:00

# Set a precise date and time using ParseExact
(Get-ChildItem file1.txt).CreationTime = [datetime]::ParseExact("07/08/2024 09:30:14", "MM/dd/yyyy HH:mm:ss", $null)
# Result: 08 July 2024 09:30:14
```

**Real-world scenarios where precise timestamps matter:**

- You are a **digital forensics expert** reconstructing a timeline of events on a computer.
- You are a **developer** testing a time-sensitive application and need files to appear as if they were created at very
  specific moments.

### Key Rule When Modifying Properties

> **Always be mindful of the data type a property expects.** If you provide a value in the wrong format, PowerShell will
> not be able to interpret your intent and the assignment will fail or produce unexpected results. Use **casting** (
`[datetime]`) or **parsing** (`[datetime]::ParseExact(...)`) to ensure PowerShell receives exactly the type it is
> expecting.

## Using Object Methods

While properties hold data, **methods perform actions**. You invoke them using the same dot notation, followed by
parentheses.

```powershell
# Copy a file to a new location using the CopyTo method
(Get-ChildItem file1.txt).CopyTo("C:\PowerShellPractice\file1_Copy.txt")

# Delete a file using the Delete method
(Get-ChildItem file1.txt).Delete()
```

Methods give you direct, object-level control over files and resources — complementing the cmdlet-based approach with
more granular, programmatic options.

---

## Working with Dates, Timestamps, and Data Types in PowerShell

When dealing with dates from different countries, timestamps with milliseconds, or any time-sensitive data, proper *
*parsing** becomes extremely important.

PowerShell is strict about data types. When modifying object properties, you must always ensure the value you provide
matches the exact type the property expects. Supplying data in the wrong format can lead to errors or unexpected
behavior.

### Why This Matters

- As a **Digital Forensics Expert**, you often need to reconstruct an accurate timeline of events from a compromised or
  analyzed computer.
- As a **Developer** working on time-sensitive applications, you may need to test how your application behaves when
  files are created or modified at very specific timestamps.

In both cases, understanding the expected data type of property is critical. Always use proper **casting** or *
*parsing** techniques to ensure PowerShell interprets your date and time values correctly.

---

**Key Takeaway:**  
Never assume PowerShell will automatically understand your date format. Be explicit — parse and cast your data properly
to avoid incorrect results or script failures.

## Command Input & Output Fundamentals

- PowerShell's revolutionary feature is that it works with objects (structured data with properties and methods), not
  plain text like traditional shells.

```powershell
# Input: Directory path → Output: File/Folder objects
Get-ChildItem C:\Windows

# Displays Notepad properties and methods
Get-Process notepad | Get-Member
```

### The Pipeline (|) – PowerShell's Superpower

The pipeline passes live objects between cmdlets, enabling powerful chaining

| cmdlet              | description                                               |
|---------------------|-----------------------------------------------------------|
| Get-Process notepad | outputs a Process object                                  |
| Get-Member          | receives that object and reveals its properties & methods |

#### Input Behavior Types

| Type               | Description                       | Example                                                               |
|--------------------|-----------------------------------|-----------------------------------------------------------------------|
| **Parameter-only** | Accepts input via parameters only | `Get-Process -Name notepad`                                           |
| **Pipeline-only**  | Works mainly with pipeline input  | `Get-Process \| Get-Member`                                           |
| **Both**           | Flexible input methods            | `Stop-Process -Name notepad` OR `Get-Process notepad \| Stop-Process` |

## Sorting and Selecting Items

```powershell
# Top 5 CPU-consuming processes
Get-Process | Sort-Object CPU -Descending | Select-Object -First 5

# Top 3 memory-heavy processes (your example)
Get-Process | Sort-Object WorkingSet -Descending | Select-Object -First 3

# Select with Calculated Properties
Get-Process | Select-Object Name,
@{ Name = 'MemoryMB'; Expression = { $_.WorkingSet64 / 1MB } },
@{ Name = 'Company'; Expression = { $_.Company?.Name ?? 'Unknown' } }

```

## Measure-Object – The Analysis Powerhouse

```powershell
# Count files/folders in current directory
Get-ChildItem | Measure-Object

# Full directory statistics
Get-ChildItem | Measure-Object -Property Length -Sum -Average -Maximum -Minimum

# Check Downloads folder size (converts to MB)
Get-ChildItem ~/Downloads -File -Recurse |
        Measure-Object -Property Length -Sum |
        Select-Object @{ N = 'TotalSizeMB'; E = { [math]::Round($_.Sum / 1MB, 2) } }

# Find suspiciously large files (>100MB)
Get-ChildItem C:\Users -File -Recurse -ErrorAction SilentlyContinue |
        Where-Object { $_.Length -gt 100MB } |
        Sort-Object Length -Descending

# Standard deviation for CPU analysis (identify outliers)
Get-Process | Measure-Object -Average -StandardDeviation CPU

# Measure all stats at once
1..100 | Measure-Object -AllStats

# ScriptBlock properties for custom calculations
Get-ChildItem | Measure-Object -Sum { $_.Length/1MB }

# Count lines, words, characters in a file
Get-Content C:\Temp\log.txt | Measure-Object -Line -Word -Character
```

## Grouping and Analyzing Data

**Group-Object for Aggregation**

- Group data by properties for deeper analysis.

```powershell
# Group processes by company
Get-Process | Group-Object -Property { $_.Company?.Name ?? 'Unknown' } |
        Select-Object Name,
        @{ N = 'ProcessCount'; E = { $_.Count } },
        @{ N = 'TotalMemoryMB'; E = { ($_.Group | Measure-Object WorkingSet64 -Sum).Sum / 1MB } }

# Group files by extension and calculate total size per type
Get-ChildItem -File | Group-Object Extension |
        ForEach-Object {
            $size = ($_.Group | Measure-Object Length -Sum).Sum
            [PSCustomObject]@{
                Extension = $_.Name
                Count = $_.Count
                TotalSizeMB = [math]::Round($size / 1MB, 2)
            }
        } | Sort-Object TotalSizeMB -Descending
```

## Formatting and Customizing Output

```powershell
Get-Process | Sort-Object CPU -Descending |
        Select-Object -First 5 Name, CPU, WorkingSet |
        Format-Table -AutoSize

# List format for detailed view
Get-Process | Format-List Name, Path, Company, CPU, WorkingSet

# table format 
Get-Process | Format-table Name, Id, CPU

# 
Get-ChildItem | Format-Wide
Get-ChildItem | Format-Wide -Column 4

# Custom Objects with ForEach-Object
Get-Process | ForEach-Object {
    [PSCustomObject]@{
        ProcessName = $_.Name
        MemoryUsageMB = [math]::Round($_.WorkingSet64 / 1MB, 2)
        CPUPercent = $_.CPU
        StartTime = $_.StartTime
    }
} | Sort-Object MemoryUsageMB -Descending
```

## Exporting and Saving Output

```powershell
# CSV for Excel/analysis
Get-Process | Select-Object Name, CPU, WorkingSet |
        Export-Csv -Path "processes.csv" -NoTypeInformation

# JSON for APIs/web
Get-Process | Select-Object -First 5 | ConvertTo-Json | Out-File processes.json

# HTML report
Get-Process | ConvertTo-Html -Property Name, CPU, WorkingSet | Out-File report.html

# XML
Get-Process | Export-Clixml processes.xml
```

## Real Power: Multi-System Management

- "real power comes when dealing with multiple systems" is crucial. Use PowerShell Remoting or CIM sessions:

```powershell
# Run on multiple computers
$computers = 'Server01', 'Server02', 'Server03'
Invoke-Command -ComputerName $computers -ScriptBlock {
    Get-Process | Sort-Object CPU -Descending | Select-Object -First 5
} | Select-Object PSComputerName, Name, CPU, WorkingSet

# CIM for non-Windows (uses standard protocols)
Get-CimInstance -ClassName Win32_Process -ComputerName $computers |
        Select-Object ProcessName, WorkingSetSize, PSComputerName
```

## Sample Script

```powershell
# System Resource Report across multiple servers
$servers = 'Web01', 'App01', 'DB01'

$report = Invoke-Command -ComputerName $servers {
    # Get top memory processes
    $topProcs = Get-Process | Sort-Object WorkingSet -Descending |
            Select-Object -First 3 Name, WorkingSet, CPU

    # Get disk usage
    $diskUsage = Get-CimInstance Win32_LogicalDisk |
            Where-Object { $_.DriveType -eq 3 } |
            Select-Object DeviceID,
            @{ N = 'FreePercent'; E = { [math]::Round(($_.FreeSpace/$_.Size)*100, 2) } }

    [PSCustomObject]@{
        Server = $env:COMPUTERNAME
        TopProcesses = $topProcs
        DiskStatus = $diskUsage
        Timestamp = Get-Date
    }
}

# Export to JSON for further processing
$report | ConvertTo-Json -Depth 3 | Out-File "ServerReport_$( Get-Date -Format yyyyMMdd ).json"
```
