## Tokens

- A `token` is the smallest chunk of the text an AI model understands.
- It can be a a full word, part of the word, punctuation, even spaces etc
- 1 token = 3 to 4 characters may be 5 to 6 sometimes. (English)
- Code = more tokens (because of symbols and formatting)

### Input Tokens vs Output Tokens

- This is where billing + performance matters.

**Input Tokens**

- Everything we send to the model.
- It can be a question, System prompt, Chat history, Documents, Context/Memory.
- Input tokens are cheaper compared to output tokens because reading is easy task.

**Output Tokens**

- Everything model generates.
- It can be the answer, code, explanations etc.
- Output tokens are expensive than input token because generating solutions is computationally heavy.

**Why tokens are important?**

1. Cost Control: If we use more tokens means we need to pay more money. If we send more longer prompts and model
   generates long responses increases bill.
2. Performance : If we use more tokens means taken more time for processing and slower responses.
3. Memory: If we use more tokens means Every model has context window. If we exceed the window size the model forget the
   older data or information.

----

## Context Window

- Context window is the maximum number of tokens the model can handle at once or in one request. Think of it like a
  temporary memory
  bucket.
- If a model has 128K tokens context window. (System prompt, Chat history, documents, question, answer). all of these
  should accommodate in 128K tokens.

**What happens when window is filled**

- When window is overflowed, this is called context overflow.
- Old messages get removed. Model starts forgetting earlier conservations.
- It is not a bug, it is a limitation.

**How to solve overflow issue**

- Memory Optimization : Old data is summarized. Instead of keeping full history.
- Retrieval Augmented Generation (RAG): Instead of sending complete documents. Search relevant parts and send only
  useful chunks.
- Limit Output Size : Control response length.
- Cache Inputs : Reuse repeated prompts and saves cost.

**Cost Optimization Strategies**

1. Choose the Right Model (Don't use GPT-4o for tasks GPT-4o mini handles well.)

```java

@Service
public class SmartChatService {

    private final ChatClient premiumClient;  // GPT-4o
    private final ChatClient budgetClient;   // GPT-4o mini

    public String chat(String question, TaskComplexity complexity) {
        return switch (complexity) {
            case SIMPLE -> budgetClient.prompt().user(question).call().content();
            case COMPLEX -> premiumClient.prompt().user(question).call().content();
        };
    }
}
```

2. Cache Responses (Same question = same answer. Don't pay twice.)

- If the same question is asked again, return the saved answer instead of calling the AI again.

```java

@Service
public class CachedChatService {

    private final ChatClient chatClient;
    private final Cache<String, String> cache;

    public String ask(String question) {
        String cached = cache.getIfPresent(question);
        if (cached != null) {
            return cached;  // Free!
        }

        String response = chatClient.prompt().user(question).call().content();
        cache.put(question, response);
        return response;
    }
}
```

3. Limit Output Tokens (Control response length to control cost.)

- Tell the model not to write too much.

```java
class LimitToken {
    ChatResponse response = chatClient.prompt()
            .user("Summarize this document")
            .options(ChatOptions.builder()
                    .maxTokens(500)  // Cap the output
                    .build())
            .call()
            .chatResponse();
}
```

4. Compress Input (Shorter prompts = fewer input tokens)

- Verbose: I would like you to please provide me with a detailed explanation of how the Spring Boot framework handles
  dependency injection in a comprehensive manner (28 tokens)
- Concise: Explain Spring Boot dependency injection (6 tokens)
- Same result, significantly cheaper on input.

5. Batching: Reduce Costs for Bulk Operations

- When you have many requests and don’t need instant answers, use batch APIs.
- For example: OpenAI Batch API gives 50% off with max wait time of 24 hours. Use cases are Nightly processing, bulk
  analysis

```java
class BatchAPI {

    public void method() {
        // Instead of 1000 individual calls
        for (Document doc : documents) {
            processDocument(doc);  // Expensive: 1000 API calls
        }

        // Batch them together
        List<BatchRequest> batch = documents.stream()
                .map(this::createBatchRequest)
                .toList();

        batchClient.submitBatch(batch);  // Cheaper: 1 batch call, 50% off
    }
}
```

**When to use Batching**

    Nightly data processing jobs
    Bulk document analysis
    Training data generation
    Non-real-time classification tasks

6. Hybrid Cloud + Local Setup

- For optimal cost and performance, combine cloud APIs with local models.
- For example: For Sensitive/regulated data use cases and Fallback when cloud is down, use Local Setup.
- Use cheap local models for simple or sensitive tasks, and use powerful cloud models only when needed.

## Vector Embeddings

**The problems with tokens**

- The model cannot understand meaning from IDs alone.
- king != monarch (different IDs). Humans know it as same but model fails using token system.
- bank (financial institution) and bank (river side). Same tokenId but meaning changes with context. Token systems gets
  confused.
- Tokens lack semantic understanding and cannot measure similarity.
- Here is where `Vector Embeddings` comes into picture.
- Embeddings means Numbers that capture meaning + context.

```text
Tokens = Syntax (What text looks like)
Embeddings = semantics (What text means)
```

**Tokens vs Embeddings**

    "king" -> 1023 (Normal Token)
    "king" -> [0.12,-0.45,0.98...1536 values] (Embeddings)

- A vector Embedding is a high dimensional numerical representation of text that captures its meaning.
- More dimensions means more semantic details.
- Words with similar meaning stays closer in space. Normally ai models use math to measure similarity.
- cosine similarity, euclidean distance.

*Consine Similary*

- A metric used to measure similarity between two vectors based on angle.

```text
 GPS -> [latitude, longitude] -> 2D (2 dimensions)
 Embeddings -> [0.25,-0.87,0.87...1536 values] -> 1536 dimensions
```

**Why embeddings are important?**
**Without Embeddings:**

- If user asks `How to fix login issue?`
- You have given an large document about your application.
- Document contains words like `Authentication troubeshooting guide.`
- There is no way your ai model understand the issue as No keyword match. Model fails to give you solution.

```text
No Similarity
No Search
No RAG
No Smart AI
```

**With Embeddings:**

- If user asks `How to fix login issue?`
- You have given an large document about your application.
- Document contains words like `Authentication troubeshooting guide.`
- The words like `login issue` and `authentication problem` are close.
- Ai model understands the problem and retrieves correct solution from the document. This is called `semantic search`
- Embeddings are the foundation of `Retrieval Augmented Generation`

```text
Convert documents -> embeddings
Store in vector DB
Convert user Query -> embedding
Find closest matches
Send relevant chunks to LLM

# OriginalFlow

UserQuery -> Convert to Embeddings -> Vector Search (Similarity) -> Retrieve top results -> Send to LLM -> Generate answer
    
```

## Transformers

- A `Transformer` is just a machine that reads text, figures out relationships between words and then produces output.
- It does not read word-by-word like old models. It process everything in parallel.
- It processes the input very fast but also needs huge compute and memory.

# RNNs/LSTMs vs Transformers

| Old Approach (RNNs / LSTMs) | Problem                                    | How Transformers Fix It                                   |
|-----------------------------|--------------------------------------------|-----------------------------------------------------------|
| Sequential processing       | Slow — cannot parallelize computations     | Self-attention processes all tokens in parallel           |
| Fixed memory                | Forgets beginning of long text             | Attention can look at any part of the sequence anytime    |
| Word-by-word reading        | Limited ability to understand full context | Considers all words at once for richer understanding      |
| Limited context window      | Struggles with long-range dependencies     | Models learn global relationships across the entire input |

### Attention

- `Attention` means deciding which words matter for understanding another word.
- `Self-Attention` means each word looks at every other word in the same sentence.

```text
“The bank was steep, so the river flowed fast.”
When understanding “bank”, the model checks:

river → high relevance
flowed → high relevance
financial → low relevance

So it doesn’t “remember everything”, it re-weights importance dynamically.
So for every word: it asks: “which other words help explain me?”
It is NOT understanding meaning directly
It is computing similarity scores between vectors (numbers)
```

### Q, K, V (Query, Key, Value)

**Query** What i am trying to understand right now?
**Key(K)** What information do i offer?
**value(V)** what actual information I carry?

- QKV is not “intelligence”, it’s a structured matching system using linear algebra.

```text
Think of YouTube search:
Query = your search text
Key = video titles/tags
Value = video content

So:
Query matches Keys
Matching decides importance
Values are combined to produce output
```

**Attention Weights**

- These are just numbers. high weight → word is important, low weight → word is ignored.

```text
“The bank was steep, so the river flowed fast.”

For “bank”:
river → 0.9
money → 0.1

These weights are learned during training.
```

### Multi-Head Attention

- Instead of one attention system, you run many in parallel.
- Each head learns a different pattern

```text
grammar relationships
meaning similarity
long-distance references
syntax structure
```

### Transformer Layers

- A transformer is repeated blocks

```text
Attention (figures relationships)
Feed-forward network (refines transformation)
Normalization (stabilizes values)
```

- Each layer builds slightly deeper understanding
- It does NOT “understand more like humans layer by layer”. It just transforms representations repeatedly.

### Feed-Forward Network (FFN)

- After attention mixes information.FFN processes each word independently
  applies non-linear transformation.

### Position Encoding

- Position Encoding means numbers added to each word in the input to show order.
- `Attention` alone sees `dog bites a man` and `man bites a dog` as same.
- So it is important to inject position information.
- Modern models often use better versions of `position Encoding` which is `rotary embeddings`.

### Encoder vs Decoder

- `Encoder` reads input, builds understanding and does not generate text.
- These are used in embeddings, search systems, BERT like models.
- `Decoder` generates text token by token and predicts the next word repeatedly.
- These are used in GPT, Claude-style chat models.
- `RAG Systems` are combinations of both Encoder and Decoder. Encoder understands documents and input text. Decoder
  generate the answer.

```text
Transformer = repeated system of vector comparisons + weighted averaging + nonlinear transformation
```

# Transformer Concepts Summary

| Concept              | Simple Takeaway / Practical Use                                                  |
|----------------------|----------------------------------------------------------------------------------|
| Attention            | Model sees all context at once – good for long prompts                           |
| Context Window       | Limited size – attention looks at all word pairs, so very long inputs are costly |
| Multi-Head Attention | Different parts of your prompt are processed for different meanings              |
| Layers               | More layers = better understanding, but slower and more expensive                |
| Positional Encoding  | Order of words matters – helps the model understand relationships                |

### How an LLM Works (Clear Learning Explanation)

An LLM (Large Language Model) is a system that learns patterns in language by training on massive amounts of text. Its
goal is not to “understand” language the way humans do, but to predict what word (or more accurately, token) should come
next in a sequence. Over time, this simple objective produces surprisingly intelligent behavior such as reasoning,
summarization, and coding ability.

The name itself is descriptive. “Large” refers to the size of the model, which can contain billions or even trillions of
parameters. These parameters are internal numerical values learned during training. “Language” means the model is
trained on human language data such as books, articles, code, and other text sources. “Model” means it is a mathematical
system that takes input and produces a prediction, primarily the next token in a sequence.

**Tokens – How Text is Broken Down**

Before an LLM processes text, it cannot directly understand raw sentences. Instead, text is broken into smaller units
called tokens. A token is not always a full word; it can be a word, part of a word, or even punctuation depending on how
the tokenizer is designed.

For example, a word like “unbelievable” may be split into multiple tokens like “un”, “believ”, and “able”. This approach
allows the model to handle a very large vocabulary efficiently and also helps it deal with new or rare words that it has
never seen before.

Tokens are important because the model does not operate on characters or words directly. Everything is converted into
tokens first, and all reasoning happens over these tokens.

**Vector Embeddings – Converting Tokens into Meaningful Numbers**

Once text is tokenized, each token is converted into a vector representation called an embedding. An embedding is a list
of numbers (often hundreds or thousands of dimensions) that represents the token in a mathematical space.

The key idea is that words with similar meanings or usage patterns end up having similar vectors. For example, the
embeddings of “king” and “queen” will be closer to each other in this space than “king” and “car”. This does not mean
the model understands meaning directly; instead, it learns statistical relationships between words based on how they
appear in data.

These embeddings become the input to the transformer model, which processes them further.

**Transformer – The Core Processing Engine**

The transformer is the main architecture that powers modern LLMs. Its job is to take all token embeddings and process
them in a way that captures relationships between all words in a sentence.

Unlike older models that processed text sequentially (one word at a time), transformers process all tokens
simultaneously. This is possible because of a mechanism called attention.

Inside a transformer, each token can interact with every other token in the input. This allows the model to understand
context even when related words are far apart in a sentence.

Each transformer layer refines the understanding of the input step by step. Early layers capture simple patterns like
grammar, while deeper layers capture more complex relationships like intent, reasoning, and structure.

**Attention – How the Model Focuses**

Attention is the most important concept in transformers. It allows the model to decide which words are important when
interpreting a specific word.

For example, in the sentence “The bank was steep near the river”, the word “bank” is ambiguous. The model uses attention
to look at surrounding words like “river” and “steep” to determine that “bank” refers to the riverbank, not a financial
institution.

Technically, attention works by comparing each token with every other token and assigning importance scores. These
scores determine how much influence each word has on another word’s understanding.

This is what allows transformers to handle long-range dependencies in text, something earlier models struggled with.

**Self-Attention – Internal Relationship Building**

Self-attention is the mechanism where each token looks at all other tokens in the same input sequence and decides how
much it should depend on them.

Every token is transformed into three representations: Query, Key, and Value. The Query represents what the token is
looking for, the Key represents what it contains, and the Value represents the actual information it carries.

By comparing Queries and Keys, the model calculates how relevant each token is to every other token. These relevance
scores are then used to combine the Values into a new representation of each token that includes contextual
understanding.

This process is repeated for every token in the sequence simultaneously.

**Multi-Head Attention – Multiple Perspectives**

Multi-head attention is an extension of self-attention where multiple attention mechanisms run in parallel. Each “head”
learns to focus on different types of relationships in the text.

One head might focus on grammar structure, another on long-distance relationships between words, and another on semantic
similarity. By combining all these perspectives, the model gets a richer understanding of the input.

This is similar to analyzing a situation from multiple viewpoints at the same time instead of relying on a single
interpretation.

**Positional Encoding – Understanding Word Order**

Since attention processes all tokens in parallel, the model does not naturally know the order of words. However, word
order is crucial in language because meaning changes with structure.

To solve this, positional encoding is added to embeddings so that each token carries information about its position in
the sequence. This allows the model to distinguish between sentences like “Dog bites man” and “Man bites dog”, which
contain the same words but completely different meanings.

Modern models often use improved versions like rotary positional embeddings, but the purpose remains the same:
preserving sequence order.

**Transformer Layers – Stacked Learning**

A transformer model is made of multiple stacked layers. Each layer contains self-attention and feed-forward networks.

As data moves through these layers, the representation of each token becomes more refined. Lower layers focus on simple
patterns, while higher layers capture abstract relationships and deeper meaning.

This stacking is what gives LLMs their ability to understand complex language structures.

**Training vs Inference**

Training is the process where the model learns from massive datasets. During training, the model is shown text and
repeatedly tries to predict the next token. When it makes mistakes, it adjusts its internal parameters using a method
called backpropagation. This process is extremely computationally expensive and can take months on large GPU clusters.

Inference is the phase where the trained model is used. In this stage, the model no longer learns; it only predicts. A
user sends a prompt, and the model generates a response token by token based on learned probabilities. This is what
happens when you use ChatGPT or similar systems.

**Why Tokens Are Used Instead of Words**

Tokens are used because they provide a flexible and efficient way to represent language. If models worked only with
words, the vocabulary would be too large and difficult to manage. Tokens allow the model to break down rare or complex
words into smaller reusable parts.

This also helps the model generalize better across different languages and domains.
