## Fraud Services

### Upstream Scoring Systems

- Customer transactions come from different scoring systems
- Scoring Systems like ( Debit card - F1 Falcon, Credit card - FiServ, Online Banking - Actimize, Phone Banking -
  Pindrop, EFDES - FLM/UAR, PREVENT - PAT case)
- F1 Falcon sends transaction details to FIL
- FiServ sends transaction details to Fraud Alerts
- Actimize, FLM sends transaction details to AIAlerts through Kafka.
- Pindrop sends transaction details to Pindrop Stream Processor.
- UAR and PAT case sends transaction details to AI Services.

#### Fraud Alerts

#### Fraud AI Rules

- Cases flow through FraudAIRules (Drools engine) for decisioning.
- This module is the Drools-based BRMS decision service. It exposes invokeFraudAIRules and rollback/loadNewRule
- it uses the CoBRRa/ArGeNT pattern — XSD schema → generated JARs → rule execution.

**Core Rules to Implement:**

- Key fraud rules include: deny a transaction if more than 3 transactions occur in less than 5 seconds for the same
  user, and deny a transaction if the amount is more than twice the average of the last 4 credit card transactions.
- Enterprise applications often use stateless sessions for performance, while stateful sessions are used in
  workflow-driven systems. In enterprise systems, Drools is usually wrapped inside a decision service layer, keeping
  controllers clean and rule execution isolated.

**The XSD / Rules Schema**

- The Complex Event Processing engine allows users to implement complex, time-based requirements on streams of data in
  concise, focused rules using advanced constructs such as the accumulate construct, time-window constructs, and
  temporal operators. Red Hat
- Your XSD schema approach is correct — when you provide XSD to the ArGeNT team, they generate JAR files with proper
  POJOs that the Drools engine understands. This keeps your data model compatible with the rules engine.

#### AI Alerts

- AIAlerts is the central orchestrator managing the case lifecycle.
- This module is mian orchestrator, which manages the complete lifecycle of case processing.
- This module has multiple thread based processors. i) Retrieval Case Processor ii) New Case Processor iii) WakeUp Case
  Processor iv) Retrial Case Processor v) AutoContact Case Processor vi) Disposition Case Processor.

1. Thread per Processor

    - Use Spring State Machine to manage this. Never allow invalid state transitions.
    - NEW → RETRIEVED → ENRICHED → RULES_EVALUATED → ELIGIBLE_FOR_AUTOCONTACT → CONTACTED → DISPOSITION_RECEIVED →
      CLOSED

2. Thread Safety

    - Use @Async with a dedicated ThreadPoolTaskExecutor per processor type
    - Configure separate thread pools so a slow middleware call doesn't block case retrieval
    - Use CompletableFuture for parallel middleware enrichment calls

3. Data Enrichment (calling Middlewares: ECPR, DCBS, CCBS, APS, MCIE, EOTP):

    - Use Circuit Breaker (Resilience4j) for each middleware call
    - Cache enrichment results using Redis with short TTL (avoid re-calling on retries)
    - If enrichment fails, don't block the case — proceed with partial data and flag it

- **Transaction data from various sources is ingested as a stream of events. Additional data such as customer
  information, geolocation, and device data can be joined with the transaction data. Once enriched, the data stream is
  fed into machine learning models that identify patterns indicative of fraudulent behavior.**

#### SMS Alerts (AUTOCONTACT)

- AutoContact/SMSAlerts notifies customers via SMS/Email/Dialer
- This module facilitates auto contact customer flow — sends SMS, Email, Dialer (MQ/ANG/FICO integration).

**Priority & Retry Logic:**

- SMS first, then Email, then Dialer — based on customer preferences
- Use exponential backoff for retries (don't hammer the gateway)
- Track delivery status — if SMS fails, fall back to Email automatically

**Rate Limiting:**

- Never send more than X notifications per customer per hour — implement Redis-based rate limiting.
- Real-time alerts can achieve sub-millisecond latency from transaction generation to notification. Stateful stream
  processing implements time windowing to detect high-frequency attacks such as more than 5 transactions per minute by a
  single user.

### Real Fraud Detection Patterns (Beyond Rules)

| Pattern Name               | Description                                             | Detection Method                       |
|----------------------------|---------------------------------------------------------|----------------------------------------|
| Velocity Fraud             | Too many transactions in a short time                   | Drools time-window rules               |
| Geographic Impossibility   | Card used in two distant locations within short time    | Location + timestamp comparison        |
| Unusual Amount             | Transaction amount 3x above customer’s average          | Statistical baseline rules             |
| Merchant Category Mismatch | Gas station card used to buy electronics                | MCC code validation rules              |
| Card Not Present (CNP)     | Online purchase from new or unknown device              | Device fingerprint + IP behavior rules |
| Round Number Testing       | $100, $500 type amounts to test stolen cards            | Pattern-matching rules                 |
| Time Anomaly               | Transactions at unusual hours (e.g., 3AM inactive acct) | Time-based Drools rule                 |
| New Device + High Amount   | First-time device with unusually large purchase         | Combined risk scoring engine           |
